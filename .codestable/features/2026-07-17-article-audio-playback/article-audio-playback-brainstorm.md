# 文章内音频播放 Brainstorm

> Stage 0 | 2026-07-17 | 下一步：design

## 想做什么、为什么

博主希望把 AI 歌曲的创作过程和最终成品放进同一篇文章，让读者在阅读复盘时直接点击播放，而不是离开博客寻找外链或单独下载文件。

讨论后将问题从“给全站增加音乐播放器”收紧为“让文章承载并播放创作成果”。当前博客的 Markdown 上传流程已经能把文章和图片作为一个 ZIP 发布，因此音频也应沿用同一发布心智。

## 考虑过的方向

### 方向 A：文章内音频作品卡片

- 音频属于文章内容，在正文对应位置展示作品信息和播放控件。
- 价值：直接服务“创作过程 + 最终成品”的叙事，范围集中，移动端和桌面端都能使用。
- 代价：需要扩展 Markdown 契约和 ZIP 文章资产处理流程。
- 结论：选定。

### 方向 B：全站悬浮播放器

- 在页面底部维持独立播放器，切换文章后继续播放。
- 价值：适合连续听歌；代价是需要全局播放状态、跨页面续播和更复杂的交互。
- 结论：否决；超出当前文章叙事场景。

### 方向 C：独立音乐作品集

- 新增音乐列表、作品详情和独立管理入口。
- 价值：适合长期经营音乐内容；代价是形成第二套内容系统。
- 结论：否决；当前没有作品库诉求。

## 已敲定的设计点

- 已确认：播放器位于文章正文内，不做全站连续播放。
- 已确认：Markdown、图片和歌曲使用同一个 ZIP 上传。
- 已确认：音频作为文章资产管理，不新增歌曲表或独立音乐管理系统。
- 已确认：使用浏览器原生音频能力承载播放行为，外层提供与博客一致的作品卡片。
- 已确认：不自动播放，页面初始只加载必要的音频元数据。
- 倾向：继续保持 Markdown 原始 HTML 禁用，通过受控的音频块语法生成安全 HTML。
- 待设计：支持的首版音频格式、具体 Markdown 语法、文件大小与错误语义。

## 选定方向与遗留问题

选择“文章内音频作品卡片”：博主上传包含 Markdown、封面和音频的 ZIP，文章正文在指定位置显示作品卡片，读者可以直接播放、暂停和拖动进度。

第一版明确不做全站悬浮播放器、跨页面连续播放、独立音乐作品集、播放列表、收藏、播放统计、服务端转码、波形图和歌词同步。design 阶段需要把音频资产契约、上传校验、安全边界和可验证的浏览器行为写清楚。
