# 详细访客分析

## 用户故事

- 作为博客管理员，我希望查看每一次公开页面访问的时间、原始 IP、URL、来源、地区、设备、操作系统、浏览器及版本，以便分析内容触达和访问环境。
- 作为博客管理员，我希望在浏览器支持时看到屏幕、视口、语言、时区、CPU、内存、触控、网络能力和高熵 Client Hints，以便排查访问兼容性。
- 作为博客管理员，我希望继续使用页面浏览量、每日访客、小时、页面和设备类别等现有聚合视图，并从聚合结果下钻到访问明细。
- 作为博客管理员，我希望统计页面对所有合法 UTF-8 percent-encoded 路径统一显示可读 Unicode，而不是只处理某个标签；例如 `/tag/%E5%B7%A5%E5%85%B7` 显示为 `/tag/工具`、`/tag/%E7%BC%96%E7%A8%8B` 显示为 `/tag/编程`，同时仍可核对各自的原始编码路径。

## 为什么需要

现有统计只保存小时桶、页面、每日 HMAC 和粗粒度设备类别，无法回答“具体哪个 IP、来自哪里、使用什么浏览器版本和设备访问了哪个页面”。仅依赖 Nginx 日志也无法得到结构化地区、浏览器解析结果或浏览器端设备上下文。

owner 于 2026-07-16 明确允许保存原始 IP、完整设备信息和逐次访客明细，因此本能力取代旧 `anonymous-access-analytics` 的匿名边界。

## 怎么解决

博客对成功的公开 HTML 页面访问建立服务端事件，保存可信原始 IP、完整公开 URL/来源、原始 User-Agent/Client Hints、结构化客户端结果和本地 GeoLite2 City 近似地区。第一方脚本使用短期单事件签名令牌，以受限、幂等的同源接口补充浏览器实际暴露的设备上下文。管理员通过现有认证查看聚合、分页明细和单条详情。

生产主机首次部署时先下载并验证第一份 GeoLite2 City 数据库，再启动详细分析；之后每周自动更新一次。更新先进入 staging 并校验，成功后原子替换，应用无需重启自动重载。失败时继续使用上一可用数据库，后台显示数据集日期、最近更新结果和 stale 状态。

## 边界

- 允许在统一保留期内明文保存原始 IP、原始 User-Agent、公开页面 URL/查询参数、Referrer 和浏览器设备上下文。
- `/auth`、`/admin`、`/api`、静态资源、bot、非 GET/HEAD、非成功 HTML 响应不采集；OAuth `code/state`、Cookie、Authorization 和已知凭据参数值永不入库。
- 地区来自本地 GeoLite2 City，是网络出口近似结果，不等同于 GPS；客户端信息可能缺失或被浏览器/用户伪造，界面必须标明来源。
- 不采集点击、滚动、表单、键盘内容或页面正文；不引入第三方分析 SDK 或请求时在线 IP 定位。
- 不创建长期 visitor/session Cookie；本期不做跨 IP 身份识别、实时在线人数、热力图、地图或导出。
- 所有原始与结构化事件字段使用同一可配置保留期；管理员页面和 API 不得被共享缓存。
- 可读路径转换统一应用于 overview、事件列表和事件详情中的全部路径，不按 `/tag/工具`、标签名或路由建立特例/白名单；聚合、筛选和排序仍使用原始编码路径。
